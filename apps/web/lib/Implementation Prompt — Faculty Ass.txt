REQUIREMENT 2 — Availability-Aware Re-assignment Dropdown
Trigger
Admin clicks re-assign on a block that already has a faculty assigned.
Availability Rule — Critical

Faculty is BUSY if assigned to ANY other session whose time window overlaps the current session's time window.

Do NOT use timeslot_id = same — breaks on partial overlaps.
Use interval overlap (same logic as EMS Exam Planning clash detection — reuse that utility):
sqlexisting_session.start_time < :current_end_time
AND existing_session.end_time > :current_start_time
API
GET /api/faculty/availability?session_id=:id
Response:
json[
  {
    "id": 2,
    "name": "Prof. Ravi Kumar",
    "department_short_name": "Mech",
    "is_available": false,
    "busy_in_blocks": ["Block 201", "Block 305"]
  }
]
busy_in_blocks is always an array. Faculty can be double-booked. Render as comma-joined: "Busy: Block 201, Block 305".
Full Availability SQL
sqlSELECT 
  f.id,
  f.name,
  d.short_name AS department_short_name,
  CASE WHEN sa.faculty_id IS NULL THEN true ELSE false END AS is_available,
  GROUP_CONCAT(rb.name SEPARATOR ', ') AS busy_in_blocks
FROM faculty f
LEFT JOIN departments d ON f.department_id = d.id
LEFT JOIN supervisor_assignments sa
  ON sa.faculty_id = f.id
  AND sa.session_id IN (
    SELECT ts.id FROM timetable_sessions ts
    WHERE ts.date = :exam_date
      AND ts.start_time < :current_end_time
      AND ts.end_time > :current_start_time
      AND ts.id != :current_session_id
  )
LEFT JOIN room_blocks rb ON sa.block_id = rb.id
GROUP BY f.id, f.name, d.short_name
ORDER BY is_available DESC, f.name ASC

REQUIREMENT 3 — Assignment Notification with Accept / Deny Flow
Overview
When admin assigns or re-assigns faculty, admin must give a written reason. Faculty receives real-time notification with that reason and must Accept or Deny. Denial requires faculty to give a reason. Admin receives denial reason in real-time. No silent failures on either side.

3D — DB Schema
sqlCREATE TABLE faculty_assignment_notifications (
  id                INT PRIMARY KEY AUTO_INCREMENT,
  session_id        INT NOT NULL,
  block_id          INT NOT NULL,
  faculty_id        INT NOT NULL,
  assigned_by       INT NOT NULL,
  assignment_reason TEXT NOT NULL,
  status            ENUM('pending','accepted','denied') DEFAULT 'pending',
  denial_reason     TEXT NULL,
  delivery_status   ENUM('sent','delivered') DEFAULT 'sent',
  sent_at           DATETIME DEFAULT NOW(),
  responded_at      DATETIME NULL,
  FOREIGN KEY (session_id)  REFERENCES timetable_sessions(id),
  FOREIGN KEY (faculty_id)  REFERENCES faculty(id),
  FOREIGN KEY (assigned_by) REFERENCES users(id)
);

ALTER TABLE supervisor_assignments
  ADD COLUMN status          ENUM('pending','accepted','denied') DEFAULT 'pending',
  ADD COLUMN notification_id INT NULL REFERENCES faculty_assignment_notifications(id);

3E — WebSocket Delivery
Use Socket.io. Rooms:

Admin joins: admin:user:{admin_id}
Faculty joins: faculty:user:{faculty_id}

EventDirectionPayloadassignment:sentServer → Faculty{ notification_id, session, block, reason, assigned_by }assignment:acceptedServer → Admin{ notification_id, faculty_id, block_id }assignment:deniedServer → Admin{ notification_id, faculty_id, block_id, denial_reason }
Delivery guarantee — implement all 4:

On emit → set delivery_status = 'sent' in DB
Faculty client ACKs on receive → delivery_status = 'delivered'
Faculty offline → notification persists in DB, shown as unread badge on next login
Admin UI polls GET /api/notifications/pending every 30s as WebSocket fallback
